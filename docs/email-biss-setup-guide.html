This guide covers everything needed to implement the Student Login Time Management solution at BISS. The solution automatically enables and disables student Microsoft 365 accounts on a daily schedule using Azure Automation and Microsoft Graph API — no additional software, no server, no per-student licenses needed.

⚙️ How the Solution Works

Every school day, two automated jobs run on a schedule:

✅ 07:55 — Enable Access Azure Automation runs Enable-StudentAccess.ps1 → Sets accountEnabled = true for all students in the student security group via Microsoft Graph API.
🚫 16:05 — Disable Access Azure Automation runs Disable-StudentAccess.ps1 → Sets accountEnabled = false + revokes all active sessions (forces immediate logout).

Weekend and holiday management is configurable. Only students in the designated Entra ID security group are affected — teachers and staff are unaffected.

📋 Setup Phases — Step by Step

📄 Phase 1 — Prepare Entra ID & App Registration (~45 min | Performed by: BISS IT or Atea with temporary access)

  1. Create a student security group in Microsoft Entra admin center:
    • Go to entra.microsoft.com → Groups → New Group
    • Type: Security, Name: e.g. STM-Students-BISS
    • Add each student account as a member (or sync from School Data Sync)
  2. Create App Registration (service account for the automation):
    • Entra admin center → App Registrations → New Registration
    • Name: STM-StudentTimeManagement, Single tenant
    • Create a Client Secret (note the value — only shown once)
  3. Grant API permissions to the App Registration:
    Permission Type Purpose
    User.ReadWrite.All Application Enable/disable student accounts
    Group.Read.All Application Read group membership to find students
    User.RevokeSessions.All Application Force immediate logout when disabling accounts

    ⚠ Requires Global Administrator to click "Grant admin consent" after adding permissions.

  4. Note down the following values (needed in Phase 2):
    • Tenant ID (from Entra → Overview)
    • Application (Client) ID (from the App Registration)
    • Client Secret value
    • Student Group Object ID

☁️ Phase 2 — Deploy Azure Automation (~60 min | Performed by: Atea)

  1. Create Resource Group in Azure Portal → Resource Groups → Create
    • Name: rg-studenttime-biss-prod, Region: Norway East
  2. Create Azure Automation Account
    • Azure Portal → Create Resource → Automation → Automation Account
    • Name: aa-studenttime-biss, Resource Group: above, Region: Norway East
    • System-assigned managed identity: On
  3. Add Automation Variables (Credentials → New Variable — mark as encrypted):
    Variable Name Type Value
    TenantId String BISS Entra Tenant ID (GUID)
    ClientId String App Registration Client ID
    ClientSecret String (Encrypted) Client Secret value from Phase 1
    StudentGroupId String Object ID of the student security group
  4. Import PowerShell runbooks (provided by Atea):
    • Enable-StudentAccess.ps1 — morning script
    • Disable-StudentAccess.ps1 — afternoon script
  5. Install required modules in Automation Account → Modules → Browse Gallery:
    • Microsoft.Graph.Authentication (latest)
    • Microsoft.Graph.Users (latest)
    • Microsoft.Graph.Groups (latest)
  6. Create Schedules (Automation Account → Schedules):
    Schedule Name Recurrence Time (Oslo)
    Enable-Students-Daily Mon–Fri, Daily 07:55
    Disable-Students-Daily Mon–Fri, Daily 16:05

    Note: Azure Automation uses UTC internally. For Norway (UTC+1 in winter, UTC+2 in summer), set schedules accordingly. Our scripts handle this automatically.

🌐 Phase 3 (Optional) — Deploy Management Portal (~45 min | Adds ~720 NOK/month — App Service B1, Norway East)

If BISS wants a visual web interface for IT staff and teachers to view status, manage students and policies — skip this phase for script-only deployment.

  1. Create App Service Plan B1 in the same Resource Group (Norway East)
  2. Create Web App (Node.js 20, resource group, B1 plan)
  3. Configure Application Settings (environment variables):
    • AZURE_TENANT_ID, AZURE_CLIENT_ID, AZURE_CLIENT_SECRET
    • STUDENT_GROUP_ID — Object ID of student group
    • GROUP_NAME_PREFIX — e.g. STM- (filters which groups are shown)
    • NEXTAUTH_SECRET — Random 32-character string (for session security)
  4. Deploy portal code via GitHub Actions (CI/CD pipeline provided by Atea)
  5. Configure RBAC — assign Admin or Viewer roles to BISS IT/management users

🧪 Phase 4 — Testing & Validation (2–5 days | Performed jointly by Atea + BISS IT)

  1. Create a test group with 2–3 test accounts (NOT real students yet)
  2. Run runbooks manually from Azure Portal → Automation Account → Runbooks → Start
  3. Verify accounts are enabled/disabled in Entra admin center
  4. Test sign-in behavior: log in as test student, trigger disable runbook, confirm forced logout
  5. Verify schedule fires automatically — check Automation Account → Jobs next morning and afternoon
  6. Monitor for 3–5 school days before moving to production group

🚀 Phase 5 — Go Live (~30 min | Performed by: BISS IT)

  1. Update StudentGroupId variable in Azure Automation to point to the real student production group
  2. Inform relevant staff (teachers, reception) of the new login time rules
  3. Enable scheduled runbooks (if paused during testing)
  4. Confirm first automatic enable/disable run in the Azure Portal job logs

⚠️ Important Notes & Limitations

  • Affects login only — not device usage: This solution disables the Microsoft 365 cloud account. Students already logged into a local Windows session (not requiring cloud auth) are not immediately affected unless session revocation is triggered.
  • Teachers/staff are NOT affected — only accounts in the designated student group are managed. Admins and staff accounts are never touched.
  • Schedules run on school days only by default (Mon–Fri). Holiday management requires either manual pause of schedules or a custom holiday calendar extension (available as an add-on).
  • Client secret must be rotated annually — After 1 year, the App Registration secret expires. We recommend a reminder in your IT calendar 1 month before expiry.
  • No Intune, no Conditional Access, no VPN required — this solution is entirely cloud-based and requires none of those.
