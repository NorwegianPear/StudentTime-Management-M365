1. What This Solution Does

Microsoft 365 does not include a native feature to restrict when students can log in. This solution fills that gap by using Azure Automation to automatically enable and disable student accounts in Microsoft Entra ID (formerly Azure Active Directory) according to a configurable time schedule.

For example, you can configure student accounts to be active only Monday–Friday, 07:30–16:30, and blocked at all other times — evenings, nights, weekends, and school holidays. When a student's account is disabled, they are immediately signed out of all Microsoft 365 services (Teams, OneDrive, Office apps), and cannot sign back in until the account is re-enabled.

Two Deployment Options

OptionDescriptionBest for
Option 1 — Script Only PowerShell runbooks in Azure Automation run on a schedule. Managed entirely via the Azure Portal or PowerShell. No browser interface for non-IT staff. Schools with dedicated IT staff comfortable using Azure Portal
Option 2 — Portal + Script Includes a Next.js web portal (hosted on Azure App Service) that allows teachers and school administrators to manage schedules, suspend students, and view audit logs from any browser — no technical skills required. Schools wanting self-service for non-IT staff (recommended)
✓ No additional Microsoft 365 licenses are required. The solution works with any existing Microsoft 365 Education, Business, or Enterprise subscription. The only cost is the Azure infrastructure (see Section 9 and Section 12).

6. Register Applications in Microsoft Entra ID

Two Entra ID app registrations are required:

  • GitHubActions-Deploy — used by the GitHub Actions CI/CD pipeline to deploy Azure infrastructure (Bicep templates). Requires Contributor role on the resource group.
  • StudentAccessAutomation — used by the Azure Automation Account (via Managed Identity, not this registration directly) to read student groups and enable/disable accounts via Microsoft Graph API.
ⓘ The Automation Account uses a System-assigned Managed Identity for Graph API calls — not an app registration secret. App registrations are used for GitHub Actions deployment authentication (OIDC) and, for Option 2, for portal authentication. This means no stored passwords or client secrets anywhere in the pipeline. Reference: Managed Identity for Azure Automation ↗

6a. Register GitHubActions-Deploy (for CI/CD deployment)

1
Open Microsoft Entra admin center Go to https://entra.microsoft.com → Applications → App registrations → New registration.
Official guide: Quickstart: Register an application ↗
2
Fill in registration details
  • Name: GitHubActions-Deploy
  • Supported account types: Accounts in this organisational directory only (Single tenant)
  • Redirect URI: leave blank
Click Register. Note the Application (client) ID and Directory (tenant) ID — you will need these in Section 8.
3
Assign Contributor role on the resource group Go to portal.azure.com → Resource Groups → rg-student-access → Access control (IAM) → Add role assignment.
  • Role: Contributor
  • Assign access to: User, group, or service principal
  • Select: GitHubActions-Deploy
Reference: Assign Azure roles ↗
⚠ Do not create client secrets for the GitHubActions-Deploy registration. Authentication is handled entirely via OIDC federation (see Section 7), which requires no stored credentials.

6b. Register Portal App (Option 2 only — for browser authentication)

1
Create a second app registration Name: StudentAccess-Portal. Supported account types: Single tenant. Under Redirect URIs, add: https://<your-app-service-name>.azurewebsites.net/api/auth/callback/azure-ad
2
Create a client secret Go to Certificates & secrets → New client secret. Set expiry to 24 months. Copy the secret value immediately — it is only shown once. Store it in Azure Key Vault or as an encrypted GitHub Actions variable.

Grant Microsoft Graph permissions Go to API permissions → Add a permission → Microsoft Graph → Delegated permissions. Add:
  • User.Read — sign in and read user profile
  • GroupMember.Read.All — read group membership
Then click Grant admin consent for [your tenant]. Reference: Permissions and consent ↗

7. Configure OIDC Federation (GitHub → Azure)

OIDC (OpenID Connect) federation allows GitHub Actions to authenticate directly to Azure using a short-lived token — no client secrets, no passwords, no credentials stored anywhere. GitHub presents a signed JWT token to Azure, and Azure validates it against the registered federated credential.

Official Microsoft guide: Use GitHub Actions to connect to Azure (OIDC) ↗
Additional reference: Configure a federated identity credential on an app registration ↗

1
Open the GitHubActions-Deploy app registration In entra.microsoft.com → App registrations → GitHubActions-Deploy → Certificates & secrets → Federated credentials → Add credential.
2
Configure the federated credential Select GitHub Actions deploying Azure resources as the scenario. Fill in:
  • Organisation: your-github-org-name
  • Repository: student-access-management
  • Entity type: Branch
  • Branch: main
  • Name: github-actions-main-branch
Click Add. This trust relationship means: only GitHub Actions workflows running on the main branch of your repository are authorised to authenticate as this application.
3
Add the credentials to your GitHub repository as Actions variables In your GitHub repository: Settings → Secrets and variables → Actions → New repository secret.
  • AZURE_CLIENT_ID — Application (client) ID of GitHubActions-Deploy
  • AZURE_TENANT_ID — Directory (tenant) ID of your Entra tenant
  • AZURE_SUBSCRIPTION_ID — Your Azure subscription ID
No password or secret is stored — only public identifiers.
4
Reference in GitHub Actions workflow (.github/workflows/deploy.yml) 
- name: Azure Login (OIDC)
  uses: azure/login@v2
  with:
    client-id: ${{ secrets.AZURE_CLIENT_ID }}
    tenant-id: ${{ secrets.AZURE_TENANT_ID }}
    subscription-id: ${{ secrets.AZURE_SUBSCRIPTION_ID }}
Full workflow reference: Configuring OpenID Connect in Azure ↗
✓ Why this is more secure than using client secrets: OIDC tokens are short-lived (valid for ~10 minutes), automatically rotated on every run, and tied to a specific repository and branch. There is nothing to rotate, leak, or expire unexpectedly. Microsoft's own documentation recommends this approach for all GitHub-to-Azure CI/CD pipelines.

10. Deploy the Management Portal Option 2 Only

The management portal is a Next.js web application deployed to Azure App Service. It authenticates users against your Entra ID (only accounts you designate as administrators can log in) and provides a browser interface for managing schedules, student suspensions, and audit logs.

1
App Service is deployed via GitHub Actions (same pipeline) The CI/CD workflow deploys the Next.js build output to App Service using the azure/webapps-deploy ↗ action. No manual steps required — pushing to main triggers a full deployment.
2
Configure Entra ID authentication on the App Service In the Azure Portal: App Service → Authentication → Add identity provider → Microsoft. Select the StudentAccess-Portal app registration created in Section 6b.
Reference: Configure Entra ID authentication on App Service ↗
3
Add portal environment variables In App Service → Configuration → Application settings, add:
  • AZURE_AD_CLIENT_ID — Client ID of StudentAccess-Portal app registration
  • AZURE_AD_CLIENT_SECRET — Client secret (mark as slot setting, encrypted at rest)
  • AZURE_AD_TENANT_ID — Your tenant ID
  • NEXTAUTH_SECRET — A randomly generated 32-byte hex string
  • NEXTAUTH_URL — e.g. https://studentaccess.azurewebsites.net
4
Designate portal administrators Portal access is role-based. Atea configures an Entra ID security group (e.g. StudentAccess-Admins). Only members of this group can log in. Teachers, school administrators, and IT staff are added to this group as needed — directly from Entra admin center, no code changes required.

11. Test and Validate

1
Create a pilot group Create a separate Entra ID security group (e.g. SG-Students-Pilot) with 3–5 test accounts (staff accounts playing the role of students). Configure the schedule for the pilot group first — not the entire student body.
2
Manually trigger the Disable runbook In Azure Portal: Automation Account → Runbooks → Disable-StudentAccess → Start. Verify the pilot accounts are set to Sign-in blocked in Entra ID. Verify audit logs appear in Log Analytics.
3
Manually trigger the Enable runbook Verify accounts are re-enabled. Confirm a pilot user can sign in successfully.
4
Validate the schedule Wait for a scheduled trigger (or set the schedule to run in 5 minutes during testing). Confirm the correct accounts are enabled/disabled at the configured times.
5
Full rollout Once pilot validation is complete, move the production student group(s) to the live schedule. Monitor job output in the Automation Account for the first 2–3 days.
